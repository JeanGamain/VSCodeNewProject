VSCode

When you clicked on 'Start Debugging' to launch Chrome, VSCode read the launch.json file in the '.vscode' folder of this project. Having 'Debugger for Chrome' extentsion installed in VSCode, hopefully launched Chrome in a debug session so that you can set breakpoints in TypeScript.

./.vscode/launch.json

{
  "version": "0.2.0",
  "configurations": [
    {
      "type": "chrome",
      "request": "launch",
      "name": "Launch Chrome Debugger",
      "port": 9222,
      "url": "http://localhost:8080/",
      "webRoot": "${workspaceRoot}",
      "sourceMaps": true,
      "timeout": 15000,
      "trace":"verbose",
      "preLaunchTask": "start_server_task"
    }
  ]
}

Notice that this extentsion is set to look for source maps. These JavaScript source maps are generated by webpack, which we will get to later. The 'preLaunchTask' starts up webpack-dev-server so that you can develop using HTTP protocal locally.
This task is also located in the .vscode folder that executes a script in the package.json file.

./package.json

},
"scripts": {
  "build": "webpack",
  "start_server": "webpack-dev-server"
},

So the task calls 'start_server' in package.json, which in-turns executes the 'webpack-dev-server' command.


webpack

The primary use of webpack is to bundle and minify your projects. It relies on loaders and plugins to make any use of it. When you project a webpack project, you will install npm packages that will populate the webpack.config file. Take a glance at the webpack config file for this project and take note of the loaders that are being used. These loaders each have a corrosponding package in the package.json file.

./webpack.config.js

const path = require('path');
const webpack = require('webpack');


module.exports = {
  entry: './src/index.ts',
  output: {
    filename: 'bundle.js',
  },
  devtool: 'source-map',
  devServer: {
    contentBase: path.resolve(__dirname, 'dist')
  },
  module: {
    rules: [
      {
        test: /\.html$/,
        exclude: /node_modules/,
        use: [{ 
          loader: 'file-loader',
          options: {
            name: '[name].[ext]',
            useRelativePath: true
          } 
        }]
      },
      {
        test: /\.tsx?$/,
        exclude: /node_modules/,
        use: [{ 
          loader: 'ts-loader'
        }]
      },
      {
        test: /\.scss$/,
        exclude: /node_modules/,
        use: [{
            loader: "style-loader"
        }, {
            loader: "css-loader", options: { sourceMap: true }
        }, {
            loader: "sass-loader", options: { sourceMap: true }
        }]
      },
      {
        test: /\.css$/,
        exclude: /node_modules/,
        use: [
          {
            loader: "style-loader"
          }, {
            loader: "css-loader", options: { sourceMap: true }
          }
        ]
      },
      {
        test: /\.(png|jpg|gif|svg|ttf|eot)$/,
        exclude: /node_modules/,
        use: [
          {
            loader: 'url-loader',
            options: {
              limit: 8192,
              fallback: 'file-loader'
            }
          }
        ]
      }
    ]
  },
  
  resolve: {
    extensions: [ ".tsx", ".ts", ".js" ]
  }
};

You might of noticed the 'ts-loader' in this file. This loader transpiles TypeScript into JavaScript.


TypeScript

Code that is in TypeScript, will get bundled into the bundle.js file. When Chrome is launched in a debug session, you for instance, put a breakpoint in index.ts file of VSCode. If it binds, refresh the debugger in VSCode and it will be caught.


Tips

When Chrome launches in a debug session, if you enable the 'Developer tools' (Crtl+Shift+I), it will kill the debug session. A simply workaround is to duplicate the tab. Right-click on the tab, select 'Duplicate' so that with this new tab you can use the 'Developer tools' while the original tab in a debug session with VSCode.
Using the Developer tools you can look around on the dev-server to have a better understanding of what webpack is doing and also to troubleshoot.
